## Immutable Content Hashing Boundary

The `PolicyInsightContent` object is the immutable, hashable portion of a policy insight. It excludes read-time freshness which varies on every request.

### Content vs. Freshness Separation

| Type                   | Hashable | Description                                                    |
| ---------------------- | -------- | -------------------------------------------------------------- |
| `PolicyInsightContent` | Yes      | Immutable policy insight output. Stable identity across reads. |
| `PolicyInsightRead`    | Partial  | Content portion is hashable; `freshness` field is excluded.    |
| `freshness`            | No       | Read-time evaluation. `ageSeconds` varies per request.         |

### Freshness Formula

```typescript
ageSeconds = floor((evaluatedAtUnixMs - asOfUnixMs) / 1000);
status = ageSeconds < stalenessThreshold ? "FRESH" : "STALE";
```

The `freshness` structure is explicitly excluded from content hashing because `ageSeconds` and `evaluatedAt` vary on every read.

### Exclusive Expiry Semantics

`expiresAt` is exclusive: an insight remains valid up to but not including the `expiresAt` timestamp. After `expiresAt`, the insight MUST be treated as expired by consumers.

## Advisory-Only Authority

Policy insights are **advisory only**. They explicitly CANNOT:

- Author policy rules directly
- Make allocation decisions autonomously
- Generate trading recommendations as final
- Cause execution actions without explicit operator confirmation

The Regime Engine retains full authority over all decisions. Policy insights provide input signals; the policy engine determines final actions.

## Enum Specifications

### marketRegime

- `UP` - Bullish trending market
- `DOWN` - Bearish trending market
- `CHOP` - Sideways/ranging market

### fundamentalRegime

- `BULLISH` - Positive fundamental outlook
- `BEARISH` - Negative fundamental outlook
- `NEUTRAL` - Mixed signals
- `UNKNOWN` - Insufficient data

### posture

- `AGGRESSIVE` - Wider ranges, higher capital deployment
- `MODERATELY_AGGRESSIVE` - Moderate risk posture
- `NEUTRAL` - Balanced approach
- `DEFENSIVE` - Narrower ranges, lower deployment
- `PAUSED` - No new positions

### recommendedAction

- `HOLD` - Maintain current state
- `MONITOR_LOWER_BOUND` - Watch lower boundary
- `MONITOR_UPPER_BOUND` - Watch upper boundary
- `EXIT_TO_USDC` - Close to stablecoin
- `EXIT_TO_SOL` - Close to SOL
- `STAND_DOWN` - Halt all position management

### riskLevel

- `NORMAL` - Standard risk parameters
- `ELEVATED` - Enhanced monitoring required
- `CRITICAL` - Immediate action may be required

### reasonCode (Ordered Priority)

Ordered by decreasing priority. First code is primary driver.

| Code                         | Description                             |
| ---------------------------- | --------------------------------------- |
| `ADVISORY_ONLY`              | Informational, no specific trigger      |
| `DATA_HARD_STALE`            | Input data is stale beyond recovery     |
| `DATA_INSUFFICIENT_SAMPLES`  | Not enough data samples for confidence  |
| `CLMM_BREACH_LOWER`          | Price breached lower CLMM boundary      |
| `CLMM_BREACH_UPPER`          | Price breached upper CLMM boundary      |
| `CHURN_STAND_DOWN_ACTIVE`    | Churn stand-down period is active       |
| `CHURN_COOLDOWN_ACTIVE`      | Churn cooldown period is active         |
| `MARKET_REGIME_UP`           | Market regime classified as UP          |
| `MARKET_REGIME_DOWN`         | Market regime classified as DOWN        |
| `MARKET_REGIME_CHOP`         | Market regime classified as CHOP        |
| `FEATURE_THRESHOLD_BREACHED` | Feature indicator exceeded threshold    |
| `CONTEXTUAL_EVIDENCE_VOTE`   | Contextual evidence supports action     |
| `RESEARCH_BRIEF_ANALYSIS`    | Research brief analysis supports action |
| `NO_ELIGIBLE_PRICE_LEVELS`   | No price levels meet criteria           |

### warningCode

## Decimal String Precision

`levels.supportsUsdcPerSol` and `levels.resistancesUsdcPerSol` are decimal strings (not numbers) to preserve exact precision:

- Pattern: `^(0|[1-9]\d*)(\.\d+)?$`
- Max length: 64 characters
- Examples: `"138.5"`, `"135.20"`, `"0.0001"`

JavaScript number comparison can corrupt decimal ordering. Always compare as strings with arbitrary precision.

## Semantic Validation Limits

### Timestamps

- `generatedAt` <= `asOf` (must be non-decreasing)
- `expiresAt` > `generatedAt` (exclusive boundary)

### Arrays

- `reasonCodes`: 1-16 items, ordered by priority
- `warnings`: 0-16 items
- `levels.supportsUsdcPerSol`: 0-16 items
- `levels.resistancesUsdcPerSol`: 0-16 items

### String Lengths

- All identifier strings: 1-128 chars
- `reasoning`: 1-2048 chars
- `warnings[].message`: 1-2048 chars
- `locator`: 1-2048 chars

## Ordering Rules

### reasonCodes Ordering

Reason codes are sorted by decreasing priority. The first code is the primary driver. This ordering is semantic and validated.

### History Items Ordering

History responses return items in reverse chronological order (newest first by `generatedAt`). This is enforced, not advisory.

### Decimal Level Ordering

Support/resistance levels should be ordered ascending. Consumers must not assume this ordering is enforced by the schema (array order is preserved), but the semantic validation may reject out-of-order entries.
